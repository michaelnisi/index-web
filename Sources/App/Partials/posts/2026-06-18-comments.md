# The Overton Window in Your Codebase

> The limits of my language are the limits of my world.

– Ludwig Wittgenstein, *Tractatus Logico-Philosophicus*

There is a specific kind of madness that sets in slowly. You are iterating on a project with an AI coding assistant, maybe Claude Code, and somewhere around the third or fourth session you notice the suggestions feel narrower. Not wrong. Not bad. Just… contained. The tool is still fluent, still fast. But something has closed.

What closed is the solution space. And the mechanism is almost embarrassingly mundane: code comments.

When Claude reads a codebase, it reads everything. Types, logic, and structure—and prose. The difference is that code is constrained by a compiler. The compiler does not care what you intended; it cares what you wrote. Comments have no such check. They are unconstrained natural language, written in the same register the model uses to reason about intent. So when Claude encounters a doc comment that says *Handles authentication state transitions*, it does not read that as a description. It reads it as a decision. A prior decision, apparently made deliberately, worth preserving.

Say you have a class that manages thread safety with a lock, and a prior session left a comment describing the lock. Claude will not propose replacing it with an actor or a serial queue. Not because it can't—it knows those patterns—but because the comment has framed the lock as the design. Alternatives that would require departing from that framing simply don't appear.

The trouble is that most of those comments were not written deliberately. They were generated—by Claude, in a previous session, as part of a documentation pass. They carry the confident, declarative voice of something that knows what it’s doing. Every function summarized, every parameter explained. Total coverage, zero signal.

Human comments are chosen. Someone decided *this* was worth capturing. That selection is the signal—it tells you what was non-obvious, contested, easy to forget. Generated comments are unchosen. They are a narration of the code as it existed at a moment, flattened into apparent authority. You can no longer tell what was intentionally documented from what was auto-described. So everything gets treated as equally authoritative.

Each subsequent session reads that accumulated prose, interprets it as design intent, and generates solutions *consistent with the description* rather than with the problem currently in front of you. Your new problem gets answered through the sediment of every prior session’s narration. The creative space narrows quietly—and it feels like the project maturing, because the outputs are coherent, because nothing breaks. Coherence and closure are easy to mistake for one another.

This is where it stops being a coding problem.

The political theorist Joseph Overton described a window of politically acceptable ideas—not the full range of possible policies, but the range that a society, at a given moment, will take seriously. Ideas outside the window are not argued against. They simply do not appear. The constraint operates through omission, which makes it nearly invisible to those inside it.

Generated documentation creates exactly this structure in a codebase. Approaches that would contradict the accumulated prose do not get proposed. Not because they are bad approaches—Claude may be fully capable of them—but because the context window has been pre-loaded with a description of the project that forecloses them. The model optimizes toward consistency with its own prior output. It is not reasoning about your problem. It is reasoning about its previous description of your problem.

The result is compounding drift—solutions that work locally but silently corrode the whole. You are not building the thing you want. You are building toward a description of a thing that was written in the past, by the tool itself, without your input on what mattered.

This is the feedback loop of insanity. The model generates a description. The description becomes context. The context shapes the next generation. The generation updates the description. Around and around, each pass more self-referential than the last. Philosophers studying echo chambers call the pathological form of this an *epistemic closed system*—not a bubble, where outside voices are merely absent, but a chamber, where the structure actively reproduces itself and discredits deviation. Your codebase, over enough iterations, starts to behave like one.

What gets suppressed—the approaches that would require breaking with the accumulated narration—is exactly the kind of lateral, unexpected thinking that makes the difference between a good solution and an obvious one.

The fix is mundane: delete the model’s comments. All of them. Your comments are chosen—write sparingly, write intentionally, write what is non-obvious. Architecture decisions belong in a separate document, explicitly framed as decisions. Then let the model reason about the actual code, unencumbered by its own prior voice.

This is not a criticism of AI coding tools. It is a description of how context works and what happens when it is left to accumulate without curation. The tool is not the problem. The unexamined feedback loop is the problem. It is always the unexamined feedback loop.

The window narrows because no one noticed it was moving.
